# Detour Planar Raycast BC Baseline

This note records the first perception-reduction baseline for the planar detour task.

## Setup

Task variant:

```text
detour_planar_raycast
```

Action:

```text
[vx_body, vy_body, yaw_rate]
```

Observation layout:

```text
body_vel_xy(2)
altitude_error(1)
z_vel(1)
sin_yaw, cos_yaw(2)
rel_goal_body_xyz(3)
previous_action(3)
raycast_distances(21)
```

This removes `rel_detour_target_body_xyz`, which was effectively a local route-planner signal in `detour_planar_local`. The policy must infer obstacle/corridor structure from low-dimensional horizontal ray distances.

The demonstration labels are still generated by a privileged scripted expert that has access to the full simulator state. This intentionally creates a teacher-student setup:

```text
teacher: privileged full-state scripted expert
student: raycast observation BC policy
```

## Interpretation

This is the first follow-up variant that is not saturated by BC. The progression is:

| Variant | Observation | BC success |
|---|---|---:|
| `detour_planar` | full state | `1.0` |
| `detour_planar_local` | local route-conditioned target | `1.0` |
| `detour_planar_raycast` | raycast obstacle perception, no local detour target | `0.84` |

The result is useful because it creates headroom for RL or data-aggregation methods while keeping the action interface and task fixed. The next experiment should not change the task yet. It should compare whether BC-initialized PPO, DAgger-style relabeling, AWAC, or IQL can improve the raycast policy from the `0.84` BC baseline.

Recommended next step:

1. inspect failure rollouts for the `0.84` BC policy
2. measure whether failures are corridor-selection, final-settling, or collision failures
3. only then choose the next learning method
